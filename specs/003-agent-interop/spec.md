# Feature Specification: エージェント間連携（Agent Interoperability）

**Feature Branch**: `003-agent-interop`
**Created**: 2026-03-18
**Status**: Draft
**Input**: User description: "現在の実装では単体のシングルエージェントとしての動作しかできないが、自身が持つ能力を他のエージェントから利用できるようにする。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 外部エージェントからコスト分析ツールを発見・利用する (Priority: P1)

AI エージェント（Claude、GPT、社内エージェントなど）のオペレーターが、cost-analyzer の能力をツールとして自身のエージェントに登録し、エージェントがユーザーの代わりにコスト分析を実行できるようにする。例えば、Claude Code のユーザーが「先月の OCI コストをサービス別に教えて」と聞くと、Claude が cost-analyzer のツールを呼び出して結果を返す。

**Why this priority**: エージェント間連携の最も基本的かつ価値の高いユースケース。これが動作しなければ他の機能は意味を持たない。

**Independent Test**: エージェントクライアントから cost-analyzer のツール一覧を取得し、コスト内訳取得ツールを呼び出して結果が返ることを確認する。

**Acceptance Scenarios**:

1. **Given** cost-analyzer のエージェント連携サーバーが起動している, **When** 外部エージェントがツール一覧を要求する, **Then** 利用可能なツール（コスト内訳取得、コスト比較、ヘルスチェック等）の名前・説明・入力スキーマが返される
2. **Given** ツール一覧を取得済みの外部エージェント, **When** 自然言語クエリを引数としてコスト分析ツールを呼び出す, **Then** 構造化されたコスト分析結果（内訳または比較）が返される
3. **Given** ツール一覧を取得済みの外部エージェント, **When** 不正な引数でツールを呼び出す, **Then** エラーの種類・メッセージ・修正ガイダンスを含む構造化エラーが返される

---

### User Story 2 - 構造化パラメータによるコスト分析（LLM不要モード） (Priority: P2)

外部エージェントが自然言語パースを介さず、直接構造化されたパラメータ（期間、サービスフィルタ、分析タイプ等）を指定してコスト分析を実行する。これにより、外部エージェント側の LLM が既にユーザーの意図をパースしている場合に、二重の LLM 呼び出しを回避し、コスト・レイテンシを削減する。

**Why this priority**: エージェント連携では呼び出し側エージェントが既に意図を理解しているケースが多く、効率的な利用に不可欠。

**Independent Test**: 構造化パラメータ（開始日、終了日、分析タイプ）を直接指定してツールを呼び出し、自然言語パース無しで結果が返ることを確認する。

**Acceptance Scenarios**:

1. **Given** エージェント連携サーバーが起動している, **When** 外部エージェントが構造化パラメータ（期間: 2026-02-01〜2026-02-28、タイプ: breakdown）でツールを呼び出す, **Then** 自然言語パース処理をスキップし、直接コストデータを取得して結果を返す
2. **Given** エージェント連携サーバーが起動している, **When** 構造化パラメータで比較分析（2つの期間指定）を要求する, **Then** 比較結果（差分、変化率、トレンドサマリー）が返される
3. **Given** エージェント連携サーバーが起動している, **When** 不完全な構造化パラメータ（終了日なし等）でツールを呼び出す, **Then** 必須パラメータと入力スキーマを含むエラーが返される

---

### User Story 3 - 利用可能なフィルタ値の動的取得 (Priority: P3)

外部エージェントが、コスト分析の前にフィルタリングに使用できる値（利用可能なサービス一覧、コンパートメント一覧）を取得する。これにより、エージェントがユーザーに正確な選択肢を提示したり、有効な値のみを使用してクエリを組み立てることができる。

**Why this priority**: データ品質の向上とユーザー体験の改善に寄与するが、コスト分析の基本動作には必須ではない。

**Independent Test**: サービス一覧取得ツールを呼び出し、OCI テナントで利用可能なサービス名のリストが返ることを確認する。

**Acceptance Scenarios**:

1. **Given** エージェント連携サーバーが起動している, **When** 外部エージェントが利用可能サービス一覧ツールを呼び出す, **Then** テナントで利用可能なサービス名のリストが返される
2. **Given** エージェント連携サーバーが起動している, **When** 外部エージェントが利用可能コンパートメント一覧ツールを呼び出す, **Then** コンパートメント名と ID のリストが返される

---

### Edge Cases

- 外部エージェントが同時に多数のツール呼び出しを行った場合、各リクエストが独立して処理される
- OCI 認証情報が未設定または期限切れの場合、ツール呼び出し時に認証エラーと設定手順のガイダンスが返される
- ネットワーク障害や OCI API のタイムアウト時に、適切なエラーとリトライ可否の情報が返される。OCI API 呼び出しのタイムアウトは 30 秒とする
- ツールの入力スキーマに存在しないパラメータが渡された場合、未知のパラメータは無視する（Pydantic `extra="ignore"`）。既知パラメータのバリデーションエラーのみを返す

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、A2A（Agent-to-Agent Protocol）を通じて自身の能力（スキル）を外部エージェントに公開しなければならない
- **FR-002**: システムは、公開する各ツールについて名前・説明文・入力パラメータのスキーマ・出力形式を提供しなければならない
- **FR-003**: システムは、自然言語クエリによるコスト分析ツールを公開しなければならない（既存のクエリ処理相当の機能）
- **FR-004**: システムは、構造化パラメータによるコスト内訳取得ツールを公開しなければならない（期間、サービスフィルタ、コンパートメントフィルタを直接指定）
- **FR-005**: システムは、構造化パラメータによるコスト比較ツールを公開しなければならない（2つの期間を直接指定）
- **FR-006**: システムは、利用可能サービス一覧取得ツールを公開しなければならない
- **FR-007**: システムは、利用可能コンパートメント一覧取得ツールを公開しなければならない
- **FR-008**: システムは、ヘルスチェックツールを公開しなければならない（OCI 接続状態の確認）
- **FR-009**: すべてのツール呼び出しのエラーレスポンスは、エラー種別・メッセージ・修正ガイダンスを含む構造化された形式でなければならない
- **FR-010**: システムは、既存の CLI および Web UI の機能を損なうことなくエージェント連携機能を追加しなければならない
- **FR-011**: システムは、エージェント連携サーバーを既存の API サーバーと同一プロセスに統合し、単一プロセスで CLI・Web UI・エージェント連携のすべてを提供しなければならない

### Key Entities

- **Agent Card**: cost-analyzer が自身の能力を外部に公開するためのメタデータ。エージェント名、説明、スキル一覧、エンドポイント情報で構成される
- **スキル（Skill）**: cost-analyzer が外部に公開する個別の能力。名前、説明、入出力スキーマで構成される（A2A プロトコルの概念）
- **タスク（Task）**: 外部エージェントからの個別のリクエストとその処理状態。送信・処理中・完了・失敗の状態遷移を持つ（A2A プロトコルの概念）
- **メッセージ（Message）**: タスク内でエージェント間で交換されるデータ単位。テキストまたは構造化データ（Artifact）を含む

## Clarifications

### Session 2026-03-18

- Q: エージェント間通信プロトコルの選定（MCP / A2A / 独自） → A: A2A（Agent-to-Agent Protocol）を採用
- Q: A2A タスク処理モデル（同期 / 非同期 / ストリーミング） → A: 同期のみ（即時レスポンス、タスク状態管理なし）
- Q: Agent Card・スキル説明の言語対応方式 → A: Agent Card・スキル説明は英語のみ、コスト分析結果は言語パラメータで日英切替
- Q: A2A エンドポイントの認証方式 → A: オプショナル API キー認証。`A2A_API_KEY` 環境変数が設定されている場合、`Authorization: Bearer <key>` または `X-API-Key: <key>` ヘッダーによる認証を要求する。未設定の場合は認証なし（ネットワークレベルでアクセス制御に委任）
- Q: OCI API 呼び出しのタイムアウト → A: 30 秒。タイムアウト時は `timeout_error` を返しリトライ可能と表示
- Q: 未知パラメータの扱い → A: Pydantic `extra="ignore"` で無視。エラーにはしない

## Assumptions

- エージェント間通信には、Google 提唱の A2A（Agent-to-Agent Protocol）を使用する
- A2A エンドポイントはオプショナルな API キー認証をサポートする。`A2A_API_KEY` 環境変数が設定されている場合、リクエストに `Authorization: Bearer <key>` または `X-API-Key: <key>` ヘッダーが必要となる（HMAC 定数時間比較で検証）。未設定の場合はネットワークレベル（localhost / VPN / ファイアウォール）でのアクセス制御に委任する。OCI リソースへのアクセス認証はサーバー側の OCI 認証情報に依存する
- Agent Card およびスキル説明は英語で公開する（エージェント間通信の共通言語）。コスト分析結果は言語パラメータ（ja/en）により日英切替可能とする
- すべてのタスクは同期的に処理され、リクエストに対して即時レスポンスを返す（非同期タスク管理・ストリーミングは対象外）
- Kubernetes 上でのサービスディスカバリ規約については [004-a2a-k8s-discovery](../004-a2a-k8s-discovery/spec.md) を参照

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 外部エージェントが、ツール発見から最初のコスト分析結果取得まで5分以内に完了できる（設定含む）
- **SC-002**: 構造化パラメータによるコスト分析が、自然言語パース経由と比較してレスポンス時間を50%以上短縮する
- **SC-003**: 公開されるすべてのツール（6種類以上）が、外部エージェントから正常に呼び出し・結果取得できる
- **SC-004**: エラー発生時の100%で、構造化エラーレスポンス（種別・メッセージ・ガイダンス）が返される
- **SC-005**: 既存の CLI・Web UI・HTTP API の全機能が、エージェント連携機能追加後も同一の動作を維持する
