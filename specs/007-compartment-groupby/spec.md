# Feature Specification: コンパートメント別集計（group_by）対応

**Feature Branch**: `007-compartment-groupby`
**Created**: 2026-03-20
**Status**: Draft
**Input**: Issue #7 — コンパートメントごとの集計に対応していない

## User Scenarios & Testing *(mandatory)*

### User Story 1 - コンパートメント別コスト内訳の表示 (Priority: P1)

ユーザーがチャットUIで「コンパートメント別のコストを教えて」と問い合わせると、コンパートメント名をキーとしたコスト内訳テーブルが表示される。現在はサービス別の結果しか返せないため、コスト管理者がコンパートメント単位での予算管理・配賦を行えない。

**Why this priority**: コンパートメント別集計は Issue #7 の直接的な要求であり、この機能がなければユーザーの基本的なニーズを満たせない。コンパートメントは OCI のリソース管理の基本単位であり、部門別・プロジェクト別のコスト可視化に不可欠。

**Independent Test**: 「コンパートメント別のコストを教えて」と入力し、コンパートメント名をキーとした集計テーブルが返ることを確認する。

**Acceptance Scenarios**:

1. **Given** チャットUIが利用可能な状態, **When** ユーザーが「コンパートメント別のコストを教えて」と入力する, **Then** コンパートメント名をキーとしたコスト内訳テーブルが表示される
2. **Given** チャットUIが利用可能な状態, **When** ユーザーが「コンパートメントごとの先月のコストは？」と入力する, **Then** 先月の期間でコンパートメント別に集計されたコスト内訳テーブルが表示される
3. **Given** チャットUIが利用可能な状態, **When** ユーザーが「コンパートメント別のコストを比較して」と入力する, **Then** コンパートメント別の期間比較テーブルが表示される

---

### User Story 2 - サービス別集計の後方互換性維持 (Priority: P1)

既存のサービス別集計クエリ（「サービス別のコストを教えて」「今月のコストは？」など）がこれまで通り正しく動作する。集計軸の指定がない場合はデフォルトでサービス別集計となる。

**Why this priority**: 既存機能の破壊は許容できない。後方互換性はP1として扱う。

**Independent Test**: 既存のサービス別クエリを実行し、従来と同じ結果が返ることを確認する。

**Acceptance Scenarios**:

1. **Given** チャットUIが利用可能な状態, **When** ユーザーが「今月のコストを教えて」と入力する, **Then** サービス別のコスト内訳テーブルが表示される（従来通り）
2. **Given** チャットUIが利用可能な状態, **When** ユーザーが「サービス別のコストを教えて」と入力する, **Then** サービス別のコスト内訳テーブルが表示される（従来通り）
3. **Given** 集計軸を指定しないクエリ, **When** システムがクエリを処理する, **Then** デフォルトでサービス別に集計される

---

### User Story 3 - フィルタリングとグルーピングの組み合わせ (Priority: P2)

ユーザーが特定のコンパートメントで絞り込みつつサービス別に集計する、または特定のサービスで絞り込みつつコンパートメント別に集計するといった、フィルタリングとグルーピングの組み合わせが正しく動作する。

**Why this priority**: フィルタリング（compartment_filter）とグルーピング（group_by）は異なる概念であり、両者の組み合わせが正しく機能することで、より柔軟なコスト分析が可能になる。

**Independent Test**: 「prodコンパートメントのサービス別コストを教えて」と入力し、prod に絞り込まれたサービス別集計が返ることを確認する。

**Acceptance Scenarios**:

1. **Given** チャットUIが利用可能な状態, **When** ユーザーが「prodコンパートメントのサービス別コストを教えて」と入力する, **Then** prod コンパートメントに絞り込まれたサービス別コスト内訳が表示される
2. **Given** チャットUIが利用可能な状態, **When** ユーザーが「コンパートメント別でCOMPUTEサービスのコストを教えて」と入力する, **Then** COMPUTEサービスに絞り込まれたコンパートメント別コスト内訳が表示される

---

### Edge Cases

- 集計軸として存在しないキーワードが指定された場合 → デフォルトのサービス別集計にフォールバックし、ユーザーに対応する集計軸がない旨を通知する
- コンパートメントが1つしかない環境でコンパートメント別集計を要求した場合 → 1行のテーブルを正常に表示する
- 「リージョン別」など現時点で未対応の集計軸を要求された場合 → 未対応である旨を通知し、サービス別集計にフォールバックする
- コンパートメント別集計とコンパートメント絞り込みを同時に指定した場合（「prodコンパートメントのコンパートメント別コスト」） → コンパートメント絞り込みを適用した上でコンパートメント別に集計する（prod配下のサブコンパートメントが複数ある場合に意味がある）

## Clarifications

### Session 2026-03-20

- Q: コンパートメント別集計テーブルの表示キーは `compartment_name`（リーフ名）と `compartment_path`（フルパス）のどちらを使用するか？ → A: 通常は `compartment_name` を表示し、同名コンパートメントが存在する場合のみ `compartment_path` で区別する

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、ユーザーの自然言語入力から集計軸（サービス別 or コンパートメント別）の意図を正しく判定できなければならない
- **FR-002**: システムは、コンパートメント名をキーとしたコスト内訳テーブルを表示できなければならない
- **FR-003**: システムは、コンパートメント別の期間比較テーブルを表示できなければならない
- **FR-004**: 集計軸が明示的に指定されない場合、システムはデフォルトでサービス別集計を行わなければならない
- **FR-005**: システムは、フィルタリング（特定コンパートメントへの絞り込み）とグルーピング（集計軸の変更）を独立して組み合わせられなければならない
- **FR-006**: テーブルの列ヘッダーは集計軸に応じて動的に切り替わらなければならない（「サービス」/「コンパートメント」）
- **FR-007**: システムは、対応していない集計軸が要求された場合、デフォルトの集計軸にフォールバックし、ユーザーに通知しなければならない
- **FR-008**: コンパートメント別集計テーブルでは通常 `compartment_name`（リーフ名）を表示キーとし、同名コンパートメントが複数存在する場合のみ `compartment_path`（フルパス）で区別しなければならない

### Key Entities

- **集計軸（Group By）**: コスト集計の分類キー。サービス名またはコンパートメント名。デフォルトはサービス。
- **コスト内訳（Cost Breakdown）**: 指定された集計軸でグループ化されたコスト情報。集計キー、金額、通貨を含む。
- **コスト比較（Cost Comparison）**: 2つの期間の集計結果を同一の集計軸で比較したもの。差分と変化率を含む。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 「コンパートメント別のコストを教えて」というクエリに対し、コンパートメント名をキーとした正しい集計結果が返る
- **SC-002**: 既存のサービス別集計クエリが100%後方互換で動作する
- **SC-003**: コンパートメント別の期間比較クエリが正しく動作する
- **SC-004**: フィルタリングとグルーピングの組み合わせクエリが正しく動作する
- **SC-005**: 対応していない集計軸の要求に対して、適切なフォールバックとユーザー通知が行われる

## Assumptions

- OCI Usage API が `group_by=["compartmentName", "currency"]` をサポートしている（Issue レポートで確認済み）
- `CostLineItem` には既に `compartment_name` / `compartment_path` フィールドが存在する（Issue レポートで確認済み）
- 本フィーチャーのスコープは「サービス別」と「コンパートメント別」の2軸のみとし、リージョン別・SKU別・タグ別は将来の拡張とする
- `group_by` のデフォルト値は `service` とし、既存動作との後方互換性を維持する
