# Feature Specification: Web UI フロントエンド

**Feature Branch**: `002-web-ui`
**Created**: 2026-03-18
**Status**: Draft
**Input**: User description: "cost-analyzerのWeb UIフロントエンドを実装する。既存のFastAPI バックエンド (POST /query, GET /health) を活用し、自然言語でOCIコストを照会できるWebインターフェースを提供する。コスト内訳テーブル、比較チャート、トレンドサマリーをリッチに表示する。日本語/英語対応。"

## User Scenarios & Testing

### User Story 1 - 自然言語でコスト内訳を照会する (Priority: P1) 🎯 MVP

ユーザーがブラウザでコスト分析画面を開き、テキスト入力欄に「先月のサービス別コストを教えて」や「Show costs for last month」と入力して送信する。システムはバックエンドAPIにクエリを送信し、サービス別コスト内訳をテーブルとして表示する。テーブルにはサービス名、金額、割合が含まれ、金額の大きい順にソートされている。合計行も表示される。

**Why this priority**: コスト内訳の表示は最も基本的かつ頻繁に使われる機能であり、これだけで十分な価値を提供できるMVPとなる。

**Independent Test**: テキスト欄にクエリを入力して送信し、コスト内訳テーブルが正しくレンダリングされることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザーがWebブラウザでトップページを開いている, **When** 「先月のサービス別コストを教えて」と入力して送信する, **Then** サービス別コスト内訳テーブルが金額降順で表示され、合計行が含まれる
2. **Given** クエリを送信中, **When** バックエンドが処理中である, **Then** ローディングインジケーターが表示され、ユーザーに処理中であることが伝わる
3. **Given** バックエンドが曖昧なクエリに対して確認を求めた場合, **When** clarification レスポンスが返される, **Then** 確認メッセージと提案候補が表示され、ユーザーが選択または再入力できる

---

### User Story 2 - コスト比較を視覚的に確認する (Priority: P2)

ユーザーが「先月と今月のコストを比較して」と入力すると、2期間のコスト比較結果が表示される。各サービスの前期・当期金額、変化額、変化率がテーブルで表示される。増加・減少がひと目で分かる視覚的な表現（色分けやアイコン）があり、トレンドサマリーが自然言語で表示される。

**Why this priority**: コスト比較はコスト管理の重要なユースケースだが、内訳表示が先にないと比較の文脈が理解しにくいため、P2とする。

**Independent Test**: 比較クエリを入力し、2期間の比較テーブルとトレンドサマリーが表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザーが比較クエリを入力した, **When** バックエンドが comparison レスポンスを返す, **Then** サービス別比較テーブル（前期・当期・変化額・変化率）が表示される
2. **Given** 比較結果が表示されている, **When** ユーザーがテーブルを確認する, **Then** 増加は赤系、減少は緑系で色分けされ、変化の方向がひと目で分かる
3. **Given** 比較結果が表示されている, **When** トレンドサマリーが存在する, **Then** 自然言語による傾向説明がテーブルの下に表示される

---

### User Story 3 - 日本語と英語を切り替える (Priority: P3)

ユーザーが画面上の言語切替トグルを操作して、UIの表示言語を日本語から英語、または英語から日本語に切り替える。UIラベル、プレースホルダーテキスト、エラーメッセージがすべて切り替わる。バックエンドへのクエリには選択された言語が `lang` パラメータとして送信される。

**Why this priority**: 日本語がデフォルトで十分な価値を提供できるため、多言語対応は追加機能としてP3とする。

**Independent Test**: 言語トグルを切り替え、UIラベルとプレースホルダーが切り替わり、バックエンドへの次のクエリに正しい `lang` パラメータが付与されることを確認する。

**Acceptance Scenarios**:

1. **Given** 日本語表示の画面, **When** ユーザーが英語に切り替える, **Then** すべてのUIラベル、プレースホルダー、ボタンテキストが英語に変わる
2. **Given** 英語表示の画面, **When** ユーザーがクエリを送信する, **Then** バックエンドへのリクエストに `lang: "en"` が付与される
3. **Given** ユーザーが言語を切り替えた, **When** ページをリロードする, **Then** 選択した言語が保持されている

---

### Edge Cases

- クエリ入力欄が空のまま送信ボタンを押した場合、送信を無効化しバリデーションメッセージを表示する
- バックエンドが 502（API エラー）を返した場合、ユーザーにリトライを促すメッセージを表示する
- バックエンドが 401（認証エラー）を返した場合、管理者への連絡を促すメッセージを表示する
- ネットワークエラーの場合、接続を確認するよう案内する
- 非常に長いクエリ（1000文字超）を入力した場合、入力を制限する
- モバイルや狭い画面で表示した場合、テーブルが水平スクロール可能になる

## Requirements

### Functional Requirements

- **FR-001**: システムはトップページにクエリ入力欄と送信ボタンを表示しなければならない
- **FR-002**: クエリ送信中はローディングインジケーターを表示し、送信ボタンを無効化しなければならない
- **FR-003**: breakdown レスポンスを受信した場合、サービス別コスト内訳テーブル（サービス名、金額、割合、順位）を表示しなければならない
- **FR-004**: テーブルの金額は通貨記号付きで桁区切りフォーマットされなければならない
- **FR-005**: comparison レスポンスを受信した場合、サービス別比較テーブル（前期・当期・変化額・変化率）を表示しなければならない
- **FR-006**: 比較テーブルの変化額・変化率は増加と減少で視覚的に区別（色分け）されなければならない
- **FR-007**: comparison レスポンスにトレンドサマリーが含まれる場合、テーブルの下に自然言語で表示しなければならない
- **FR-008**: clarification レスポンスを受信した場合、確認メッセージと提案候補を表示しなければならない
- **FR-009**: error レスポンスを受信した場合、エラーの種類に応じた分かりやすいメッセージとガイダンスを表示しなければならない
- **FR-010**: UIの表示言語を日本語と英語で切り替えられるトグルを提供しなければならない
- **FR-011**: 選択された言語設定はブラウザに保存され、再訪問時に復元されなければならない
- **FR-012**: クエリ入力欄が空の場合、送信ボタンを無効化しなければならない
- **FR-013**: ヘルスチェックエンドポイントを定期的にチェックし、バックエンドの接続状態を表示しなければならない

### Key Entities

- **クエリ入力**: ユーザーが入力する自然言語テキスト。言語設定と組み合わせてバックエンドに送信される
- **コスト内訳結果**: サービス名、金額、割合、順位を持つコスト項目のリスト。期間と通貨情報を含む
- **コスト比較結果**: 2期間のサービス別比較データ。変化額・変化率・トレンドサマリーを含む
- **UIテキストリソース**: 日本語/英語のラベル、プレースホルダー、エラーメッセージ等の翻訳リソース

## Success Criteria

### Measurable Outcomes

- **SC-001**: ユーザーがクエリを入力してから結果テーブルが表示されるまで、体感待ち時間が明確にフィードバックされる（ローディング表示が即座に開始される）
- **SC-002**: コスト内訳テーブルの金額合計がバックエンドの `total` フィールドと一致する
- **SC-003**: 比較テーブルの増加・減少が100%正しく色分けされる（増加=赤系、減少=緑系）
- **SC-004**: 日本語・英語の切替が全UIラベルに対して漏れなく反映される
- **SC-005**: モバイル幅（360px以上）でもテーブルが閲覧可能である（水平スクロールまたはレスポンシブ対応）
- **SC-006**: バックエンドの全レスポンスタイプ（breakdown, comparison, clarification, error）が正しくレンダリングされる

## Clarifications

### Session 2026-03-18

- Q: フロントエンドの配信方式は？ → A: FastAPI に静的ファイルとして組み込む（同一コンテナ、追加デプロイ不要）

## Assumptions

- バックエンドAPIは既にデプロイ済みで、同一クラスタ内からアクセス可能
- フロントエンドは既存の FastAPI アプリケーションに静的ファイルとして組み込み、同一コンテナから配信する（追加のコンテナやサービスは不要）
- 認証・認可はバックエンドが担当しており、フロントエンド側での認証は不要
- ブラウザの localStorage を言語設定の永続化に使用する
- サンプルクエリ（例: 「先月のコストを教えて」）をプレースホルダーとして表示することで、ユーザーの入力を支援する

## Scope Boundaries

### In Scope

- クエリ入力と結果表示の基本UI
- コスト内訳テーブル表示
- コスト比較テーブルとトレンドサマリー表示
- エラー・確認メッセージのハンドリング
- 日本語/英語の言語切替
- レスポンシブデザイン

### Out of Scope

- ユーザー認証・ログイン画面
- クエリ履歴の保存・管理
- データのエクスポート（CSV/PDF）
- リアルタイム通知・アラート
- ダッシュボード（複数ウィジェットの同時表示）
- グラフ・チャートによる可視化（テーブル表示を優先）
