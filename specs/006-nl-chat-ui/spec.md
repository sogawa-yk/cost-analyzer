# Feature Specification: NL チャット UI

**Feature Branch**: `006-nl-chat-ui`
**Created**: 2026-03-20
**Status**: Draft
**Input**: 現在のUIは単発の自然言語問い合わせを前提としたものですが、これをチャット形式のインターフェースに変更します。結果が表形式などを返す場合に、綺麗に整形して表示できるようにします。また、結果のみを無機質に返すのではなく、ユーザーと対話するようにします。

## Clarifications

### Session 2026-03-20

- Q: 対話的応答の生成方式（テンプレート vs LLM vs ハイブリッド） → A: LLM生成 — 既存のLLMコールを拡張し、データ結果を踏まえた自然な応答文を生成する
- Q: チャット初期状態の表示内容 → A: ウェルカムメッセージ＋質問例 — システムバブルで挨拶と使い方の例をクリック可能なサジェスト付きで表示する
- Q: 処理中の追加メッセージ送信の扱い → A: 入力無効化 — 応答が返るまで送信ボタンと入力欄を無効化する

## User Scenarios & Testing *(mandatory)*

### User Story 1 - チャット形式でコスト問い合わせを行う (Priority: P1)

ユーザーはチャットウィンドウにメッセージを入力し、送信する。システムはチャット形式で応答を返し、会話の履歴が画面上に時系列で表示される。ユーザーメッセージは右側に、システム応答は左側に表示される。従来の「入力→結果表示」という単方向フローから、連続的な対話フローに変わる。

**Why this priority**: チャット形式のインターフェースへの変更が本機能の核心であり、これなしには他のストーリーが成立しない。

**Independent Test**: チャット画面を開き、「先月のコストを教えて」と入力して送信し、ユーザーメッセージとシステム応答が会話形式で表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザーがチャット画面を開いた状態, **When** テキストを入力して送信する, **Then** ユーザーのメッセージが会話エリアの右側に表示され、システムの応答が左側に表示される
2. **Given** 既に1つ以上の会話がある状態, **When** 追加の質問を送信する, **Then** 新しいメッセージが既存の会話の下に追加され、過去の会話も画面上に残っている
3. **Given** メッセージを送信した直後, **When** システムが処理中, **Then** システム側にローディングインジケーター（タイピングアニメーションなど）が表示される
4. **Given** チャット画面にメッセージが多数ある状態, **When** 新しいメッセージが追加される, **Then** 自動的に最新メッセージまでスクロールする

---

### User Story 2 - テーブル結果をチャット内で美しく表示する (Priority: P1)

システムがコスト内訳や比較結果を返す際、チャットの応答バブル内にテーブルが整形されて表示される。テーブルは通貨フォーマット、パーセンテージ、色分け（増減表示）が適用され、レスポンシブ対応で小さい画面でも読みやすい。

**Why this priority**: 結果の可読性はチャット形式と同等に重要であり、テーブルが崩れると利用価値が大幅に低下する。

**Independent Test**: 「先月と今月のコストを比較して」と送信し、比較テーブルが増減の色分け・通貨フォーマット付きでチャットバブル内に表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** 内訳クエリを送信した状態, **When** システムが結果を返す, **Then** サービス別コストテーブルが通貨フォーマット・パーセンテージ付きでチャットバブル内に表示される
2. **Given** 比較クエリを送信した状態, **When** システムが結果を返す, **Then** 比較テーブルが増加（赤）・減少（緑）の色分け付きで表示され、トレンドサマリーテキストも含まれる
3. **Given** モバイル端末でチャット画面を開いた状態, **When** テーブル結果が表示される, **Then** テーブルは横スクロールまたはレスポンシブレイアウトで読みやすく表示される

---

### User Story 3 - 対話的な応答でユーザーと会話する (Priority: P2)

システムは結果のデータだけでなく、自然な対話文を添えて応答する。例えば「先月のコスト内訳はこちらです。Compute が全体の45%を占めています。」のように、データの要約や気づきを含むメッセージとともにテーブルを返す。また、確認が必要な場合は対話的に質問を返す。

**Why this priority**: 対話的な応答はユーザー体験を大幅に向上させるが、チャットUIとテーブル表示が先に動作している必要がある。

**Independent Test**: 「先月のコストを教えて」と送信し、テーブルだけでなく要約テキスト（例：「最もコストが高いサービスは○○です」）が応答に含まれることを確認する。

**Acceptance Scenarios**:

1. **Given** コスト内訳クエリを送信した状態, **When** システムが結果を返す, **Then** テーブルに加えて、結果を要約する自然な対話文が表示される
2. **Given** 比較クエリを送信した状態, **When** システムが結果を返す, **Then** テーブルに加えて、増減の傾向や注目すべき変化を説明する対話文が表示される
3. **Given** 曖昧なクエリを送信した状態, **When** システムが確認を求める, **Then** チャットバブル内に確認の質問が自然な文章で表示され、ユーザーはそのまま回答を入力できる

---

### User Story 4 - 会話履歴をセッション内で保持する (Priority: P2)

同一セッション内での会話履歴が保持され、ユーザーは過去のやり取りを遡って確認できる。ページを離れるまで（またはユーザーが明示的にクリアするまで）、会話はブラウザ上で維持される。

**Why this priority**: 会話履歴の保持はチャットUIの基本的な期待機能だが、最小限の動作は履歴なしでも成立する。

**Independent Test**: 3回連続で質問を送信し、すべての会話が画面上に残っていること、上にスクロールして過去の会話を確認できることを検証する。

**Acceptance Scenarios**:

1. **Given** 複数回のやり取りが行われた状態, **When** 画面を上にスクロールする, **Then** 過去のすべてのメッセージとその応答が時系列で確認できる
2. **Given** 会話が存在する状態, **When** ユーザーが「会話をクリア」操作を行う, **Then** 画面がリセットされ、初期状態に戻る

---

### User Story 5 - エラー時に対話的にガイドする (Priority: P3)

エラーが発生した場合、無機質なエラーメッセージではなく、チャットの流れの中で自然にガイダンスを提供する。サジェスト例やリトライの案内を対話形式で行う。

**Why this priority**: エラーハンドリングの改善はUX向上に寄与するが、正常系が先に動作している必要がある。

**Independent Test**: 意図的に無効なクエリ（例：「天気を教えて」）を送信し、エラーメッセージがチャット形式で表示され、使い方のサジェストが提示されることを確認する。

**Acceptance Scenarios**:

1. **Given** サポート外のクエリを送信した状態, **When** システムがエラーを返す, **Then** チャットバブル内にフレンドリーなエラーメッセージと使い方の例が表示される
2. **Given** バックエンド接続エラーが発生した状態, **When** クエリを送信する, **Then** 接続状況を説明するメッセージとリトライの案内がチャット形式で表示される

---

### Edge Cases

- 非常に長いクエリ文を入力した場合、チャットバブルが適切に折り返し表示されるか？
- テーブルの列数やデータ量が多い場合（例：20サービス以上）、チャットバブル内で読みやすく表示されるか？
- 処理中は入力が無効化されるため連続送信は発生しないが、無効化の解除タイミングが応答表示完了と正しく同期しているか？
- ネットワーク遅延やタイムアウト時に、ローディング状態が適切にハンドリングされるか？
- ブラウザのページリロード時、会話履歴はクリアされる（想定動作）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、ユーザーメッセージとシステム応答を時系列のチャット形式で表示しなければならない
- **FR-002**: ユーザーメッセージは視覚的に右寄せ、システム応答は左寄せで区別して表示しなければならない
- **FR-003**: システム応答にコスト内訳テーブルが含まれる場合、通貨フォーマット・パーセンテージ・順位を含む整形済みテーブルをチャットバブル内に表示しなければならない
- **FR-004**: システム応答にコスト比較テーブルが含まれる場合、増減の色分け（増加：赤系、減少：緑系）とトレンドサマリーを含むテーブルをチャットバブル内に表示しなければならない
- **FR-005**: システムはデータ結果に加えて、結果を要約・解説する自然な対話文を応答に含めなければならない
- **FR-006**: システム処理中はシステム側にローディングインジケーター（タイピングアニメーションなど）を表示しなければならない
- **FR-007**: 新しいメッセージが追加された際、自動的に最新メッセージの位置までスクロールしなければならない
- **FR-008**: 会話履歴はブラウザセッション内（ページ遷移しない限り）で保持されなければならない
- **FR-009**: ユーザーが会話履歴をクリアする手段を提供しなければならない
- **FR-010**: エラー発生時、チャットの流れの中でフレンドリーなメッセージとサジェストを表示しなければならない
- **FR-011**: 確認が必要なクエリに対して、チャットの流れの中で確認質問を表示し、ユーザーがそのまま回答できるようにしなければならない
- **FR-012**: テーブル表示はモバイル画面でも読みやすいレスポンシブ対応でなければならない
- **FR-013**: 既存の多言語対応（日本語・英語）がチャットUI内でも維持されなければならない
- **FR-014**: チャット画面の初期表示時、システムバブルでウェルカムメッセージと質問例（クリック可能なサジェスト）を表示しなければならない
- **FR-015**: システムが応答を処理中は、送信ボタンと入力欄を無効化し、追加メッセージの送信を防止しなければならない

### Key Entities

- **メッセージ**: 送信者（ユーザー/システム）、内容（テキスト、テーブルデータ、エラー情報）、タイムスタンプ、メッセージ種別（テキスト、内訳結果、比較結果、確認、エラー）
- **会話**: メッセージの時系列コレクション。セッション内で保持され、クリア操作でリセットされる

## Assumptions

- 会話履歴はブラウザのメモリ上（JavaScript の状態）でのみ保持し、サーバー側やローカルストレージへの永続化は行わない。ページリロードで会話はクリアされる
- 対話的な応答文（要約・解説）は、既存のLLMコール（クエリパース時に使用）を拡張し、データ結果を踏まえた自然な応答文をLLMで生成する
- 既存の `/query` エンドポイントのレスポンス形式は維持し、フロントエンドでチャット形式に変換する
- 1セッション内のメッセージ数に上限は設けないが、パフォーマンス上の問題が発生しない範囲（想定：100メッセージ以内）を前提とする

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーが質問を送信してから応答が画面に表示されるまでの体験が、既存UIと同等以下の待ち時間である（チャットUI変換によるオーバーヘッドが体感されない）
- **SC-002**: 1回のセッションで5回以上の連続問い合わせを行った際、すべての会話履歴が正しく表示され、過去のテーブル結果も崩れずに閲覧できる
- **SC-003**: 内訳テーブル・比較テーブルがチャットバブル内で正しくフォーマットされ、通貨・パーセンテージ・色分けが従来と同等に表示される
- **SC-004**: モバイル画面（幅375px以上）でチャットUI全体が操作可能であり、テーブルが読みやすく表示される
- **SC-005**: ユーザーが対話的な応答から結果の要点を理解でき、テーブルだけを読む場合に比べてデータの理解が容易になる
