# 機能仕様書: NL クエリ品質改善 — E2E テスト不合格項目の修正

**機能ブランチ**: `005-nl-query-quality-fix`
**作成日**: 2026-03-20
**ステータス**: ドラフト
**入力**: E2E テストレポート（2026-03-20 実施、23件中8件 WARN）に基づく品質改善。比較クエリ・サービスフィルタ・エッジケースの3領域を修正し、全体 PASS 率を 65% → 90% 以上に引き上げる。

## ユーザーシナリオ & テスト *(必須)*

### ユーザーストーリー 1 - 比較クエリが正しく動作する (優先度: P1)

クラウド運用エンジニアが「先月と今月のコストを比較して」や「Compare costs between January and February 2026」のような比較クエリを入力した際、システムは比較に必要な2つの期間を正しく構造化し、サービスごとの差分・変化率を含む比較結果を返す。現状では比較クエリが 0/5（0%）で全滅しており、機能として事実上動作していない。

**この優先度の理由**: 比較クエリは仕様上の P2 機能であり、5件すべてが失敗している。根本原因が特定済み（パーサーが比較日付を構造化できない）であり、修正のインパクトが最も大きい。

**独立テスト**: 「先月と今月のコストを比較して」「前月比でコストはどうなった？」「Compare costs between January and February 2026」などの比較クエリを発行し、比較結果テーブルが返されることを検証する。

**受入シナリオ**:

1. **前提** ユーザーが「先月と今月のコストを比較して」と入力した場合、
   **操作** システムがクエリを処理すると、
   **結果** 先月と今月の2期間が正しく解決され、サービスごとの差分・変化率を含む比較テーブルが返される。

2. **前提** ユーザーが「前月比でコストはどうなった？」と入力した場合、
   **操作** システムがクエリを処理すると、
   **結果** 暗黙の比較対象（今月 vs 先月）が推定され、比較結果が返される。

3. **前提** ユーザーが「1月と2月を比較して」のように基準期間が曖昧な場合、
   **操作** システムがクエリを処理すると、
   **結果** いずれかを基準としてシステムが比較結果を返す、または確認質問を返す（どちらも正当な応答）。

4. **前提** ユーザーが「Compare costs between January and February 2026」と入力した場合、
   **操作** システムがクエリを処理すると、
   **結果** 2026年1月と2月の比較結果が返される。

5. **前提** ユーザーが「How do this month's costs compare to last month?」と入力した場合、
   **操作** システムがクエリを処理すると、
   **結果** 今月と先月の比較結果が返される。

---

### ユーザーストーリー 2 - コスト無関係・曖昧入力のエラーハンドリング (優先度: P2)

ユーザーがコストに無関係な入力（例:「今日の天気は？」「Hello, what can you do?」）や期間を含まない曖昧な入力（例:「コストを教えて」）を行った場合、システムはクラッシュや `api_error` ではなく、適切なメッセージを返す。現状では 3/5 件のエッジケースで `api_error`（NoneType エラーや日付バリデーションエラー）が発生している。

**この優先度の理由**: エラー時にシステムが `api_error` を返すのはユーザー体験上最悪のケース。仕様 FR-011 では「クエリ例を提供しなければならない」と定められており、明確な仕様違反である。

**独立テスト**: コスト無関係の入力や曖昧な入力を送信し、`api_error` ではなく「クエリ例」または「確認質問」が返されることを検証する。

**受入シナリオ**:

1. **前提** ユーザーが「コストを教えて」（期間なし）と入力した場合、
   **操作** システムがクエリを処理すると、
   **結果** 確認質問（「どの期間のコストですか？」）またはデフォルト期間（今月）での結果が返される。`api_error` は返されない。

2. **前提** ユーザーが「Hello, what can you do?」と入力した場合、
   **操作** システムがクエリを処理すると、
   **結果** システムの機能説明とクエリ例が返される。`api_error` は返されない。

3. **前提** ユーザーが「今日の天気は？」と入力した場合、
   **操作** システムがクエリを処理すると、
   **結果** コスト関連のクエリ例を含むガイダンスメッセージが返される。`api_error` は返されない。

---

### ユーザーストーリー 3 - サービスフィルタが安定動作する (優先度: P3)

ユーザーが「先月の Object Storage のコストは？」や「先月の Database のコストを教えて」のようにサービスを指定してフィルタリングしたい場合、システムは該当サービスのコストのみを返す。現状では「Compute」はフィルタが効くが、「Object Storage」「Database」では全サービスが返される不安定な状態。

**この優先度の理由**: フィルタが効かないとユーザーは結果を目視でフィルタリングする必要があり、ツールの価値が大幅に低下する。ただし P1/P2 と異なり「動作はするが精度が低い」状態であり、致命的ではない。

**独立テスト**: 「先月の Object Storage のコストは？」「先月の Database のコストを教えて」などのサービスフィルタ付きクエリを発行し、指定サービスのコストのみが結果に含まれることを検証する。

**受入シナリオ**:

1. **前提** ユーザーが「先月の Object Storage のコストは？」と入力した場合、
   **操作** システムがクエリを処理すると、
   **結果** Object Storage のコストのみが返される（全サービスは返されない）。

2. **前提** ユーザーが「先月の Database のコストを教えて」と入力した場合、
   **操作** システムがクエリを処理すると、
   **結果** Database 関連のコストのみが返される。

3. **前提** ユーザーが「Show only Object Storage costs for last month」と入力した場合、
   **操作** システムがクエリを処理すると、
   **結果** Object Storage のコストのみが返される。

---

### エッジケース

- パーサーが比較日付（`comparison_start_date`/`comparison_end_date`）を省略した場合: バリデーションエラーではなく、フォールバック処理（自動推定または確認質問）が実行される。
- パーサーが無効な日付（`start_date >= end_date`）を返した場合: `api_error` ではなく、デフォルト期間の適用または確認質問が返される。
- パーサーが `NoneType` を返した場合: `api_error` ではなく、クエリ例を含むガイダンスメッセージが返される。
- パーサーがサービスフィルタを省略した場合（ユーザーはサービスを指定した）: フォールバックとして全サービスの結果を返しつつ、フィルタが適用されなかった旨を通知する。

## 要件 *(必須)*

### 機能要件

- **FR-001**: システムは比較クエリ（`query_type: COMPARISON`）において、2つの期間（基準期間と比較期間）を正しく構造化しなければならない (MUST)。パーサーが比較日付を省略した場合、フォールバック処理を実行しなければならない。
- **FR-002**: フォールバック処理は以下のいずれかでなければならない (MUST): (a) クエリ文脈から比較日付を自動推定する、(b) ユーザーに確認質問を返す。
- **FR-003**: システムはパーサーからの不正出力（`NoneType`、`start_date >= end_date`、必須フィールドの欠落）を捕捉し、`api_error` ではなくユーザーフレンドリーな応答を返さなければならない (MUST)。
- **FR-004**: コスト無関係の入力に対して、システムはクエリ例を含むガイダンスメッセージを返さなければならない (MUST)（既存仕様 FR-011 の実装修正）。
- **FR-005**: 期間を含まない曖昧な入力に対して、システムは確認質問またはデフォルト期間（今月）での結果を返さなければならない (MUST)。
- **FR-006**: システムはサービスフィルタ付きクエリにおいて、指定されたサービス名を安定的に抽出しなければならない (MUST)。少なくとも OCI の主要サービス（Compute、Object Storage、Database、Networking、Block Volume、Functions 等）をカバーすること。
- **FR-007**: サービスフィルタの抽出精度は、テスト対象サービスの 90% 以上で正しくフィルタが適用されなければならない (MUST)。

### 前提条件

- 本機能は既存の NL コストクエリ機能（001-nl-cost-query）の品質改善であり、新機能の追加ではない。
- パーサーのプロンプト改善とバリデーションロジックの強化が主な対処方針である。
- OCI コスト管理 API の仕様や認証方式に変更はない。

## 成功基準 *(必須)*

### 測定可能な成果

- **SC-001**: E2E テストスイート（23件）の PASS 率が 90% 以上（21件以上 PASS）に到達すること。改善前は 65%（15/23 PASS）。
- **SC-002**: 比較クエリ 5件中 4件以上が PASS すること。改善前は 0/5。
- **SC-003**: エッジケース 5件中 4件以上が PASS すること。改善前は 2/5。
- **SC-004**: サービスフィルタ 3件中 3件が PASS すること。改善前は 1/3。
- **SC-005**: 既存の内訳クエリ（日本語5件・英語5件）が退行しないこと（10/10 PASS を維持）。
- **SC-006**: すべてのエラーケースにおいて `api_error` が返されないこと。代わりにクエリ例、確認質問、またはガイダンスメッセージが返されること。
